## 📌 概要
本プロジェクト（通称：DPT）は、運送・物流の現場（トラックドライバー・自販機ルートマン）で長年ハンドルを握ってきた経験（ドメイン知識）と、Webフロントエンドの基礎技術を掛け合わせて開発した **「運転性格診断クイズ（全50問）」** です。

一般的なクイズアプリにある「正解か不正解か」という単純な2値化ロジックではなく、現場のリアルな心理や他者依存のリスクに応じた **「安全性のグラデーション（5段階の傾斜配点アルゴリズム）」** をJavaScriptで実装しています。法律上の正論だけでは割り切れない現場の葛藤やリスク許容度を、自分自身の運転感覚と照らし合わせて客観的に振り返ることができるツールです。

## 🌟 主な機能と特徴
1. **5段階の傾斜配点データ構造**
   - 全50問すべての選択肢に `2.0点` / `1.5点` / `1.0点` / `0.5点` / `0.0点` の5段階評価を定義。データ構造（`test_data.js`）の対称性を意識し、最終的に0点〜100点満点で算出される加算式ロジックを構築しています。
2. **5大運転性格ジャンルによるプロファイリング**
   - 算出された合計得点に基づき、現場のドライバー特性を捉えた5つのタイプに判定し、詳細な解説文を出力します。
     - 🟢 **【安全第一】** 堅実な防衛運転タイプ
     - 🔵 **【チームプレイヤー】** 協調・マナー重視タイプ
     - ⚪ **【ポーカーフェイス】** 冷静沈着な鉄のメンタルタイプ
     - 🟡 **【タイムキーパー】** 効率重視のベテランタイプ
     - 🔴 **【ハイリスク】** 感情優先のイケイケタイプ
3. **過去の受診履歴管理（データの永続化）**
   - `sql.js` とブラウザストレージを繋ぎこみ、ページをリロードしても過去の診断スコアと受診日時が消えずに保持されます。
4. **軽量・クリーンな構成 & 拡張性**
   - 外部ライブラリ（jQuery等）に依存しないバニラJavaScript（Vanilla JS）で実装。設問データを独立したオブジェクト配列として分離しており、将来的なJSON外部化やバックエンドAPI（Flask/Django等）からの非同期取得（Fetch API）へ即座に対応可能です。

## 📊 診断カテゴリ（全50問）

運転性格を「外面的なリスクへの対応」と「内面的なリソースの制御」の両面から捉えられるよう、5つのカテゴリに分類し、全方位のリスクを網羅しています。
- **公道での他車・歩行者との遭遇** (10問)：狭路での対向車待避、バスの発進予測、割り込み対応など
- **運行管理とスケジュール圧迫** (10問)：出発遅れのリカバリー判断、430規制と渋滞の板挟み、荷主からの時間短縮要求への対応など
- **現場・荷役・バース着け** (10問)：死角の多いバック進入、手積み・手降ろしの安全確保、輪止め設置の徹底など
- **異常気象と物理的限界** (10問)：ゲリラ豪雨、ブラックアイスバーン、強風下の高架橋、アンダーパス冠水時の判断など
- **プロとしての精神力と他車への配慮** (10問)：煽り運転への感情処理、緊急車両への進路の譲り方、割り込みへの反応、疲労時の確認動作の質、単調な長時間走行での集中力維持など

## 🔍 技術的アプローチと最適化（こだわったポイント）

### 1. ブラウザ完結型SQLiteの採用とリソース管理
本アプリはサーバーレス（GitHub Pages等による静的ホスティング）での軽量な実行を前提としつつ、本格的な関係系データベース（RDB）による履歴管理を疑似体験するため、WebAssemblyで動作する `sql.js` を採用しました。

### 2. データの永続化と技術的トレードオフの割り切り
初期実装では `sql.js` を純粋なインメモリ（RAM上）で起動していたため、ページのリロードや離脱によってデータが消失する課題がありました。これを解決するため、**「SQLの実行状態（バイナリ）をLocalStorageへ退避・復元する」**アーキテクチャを構築しました。

- **課題とトレードオフの考察**:
  データを `INSERT` するたびに、データベース全体のバイナリ（Uint8Array）をJSON配列へ変換して `LocalStorage` へ同期（書き込み）するため、将来的にレコード数が極限まで肥大化した場合にはI/Oのボトルネックになるリスク（技術的負債）を認識しています。
- **現フェーズでの設計判断（事実ベース）**:
  本アプリが管理するデータは「スコア（数値）」と「受診日時（文字列）」のみであり、1レコードあたり数十バイト程度です。100回以上の受診を重ねても数キロバイトのデータ量にとどまるため、`LocalStorage` の容量制限（約5MB）やブラウザのメインスレッドへの負荷に対して、体感できるパフォーマンス低下は一切発生しないと計算し、現行のシンプルな設計を採用しました。
- **次フェーズへのロードマップ**:
  今後、1問ごとの詳細ログなど大容量データを扱う場合は、純粋なJSON配列としての個別保存、または非同期I/Oが可能な `IndexedDB` への移行による最適化を行います。

### 3. 他アプリケーションとのデータ衝突（競合）回避
同一ブラウザ内で他のローカルアプリケーションが起動しているケースを想定し、LocalStorageのキー名をアプリ固有のユニークな名称（`truck_driver_test_db_v1`）に設計。キーの重複によるデータの汚染や意図しない上書きリスクを完全に排除しています。
